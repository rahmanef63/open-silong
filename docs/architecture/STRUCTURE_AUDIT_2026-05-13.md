### Fix Strategy by Pattern

#### Pattern 1 — Bidirectional `editor ↔ databases` (8 edges)

```
editor → databases  (5x)
databases → editor  (3x)
```

The two slices co-render each other's content. Editor displays database blocks; database cells host editor for rich text.

**Fix:** Introduce a `shared/block-renderer/` slice with:

- `BlockRendererContract` — interface a slice implements to participate in block rendering.
- `BlockRendererRegistry` — runtime map slice-name → renderer.
- Editor registers its block renderer in `editor/init.ts`.
- Databases registers its renderer in `databases/init.ts`.
- Each slice imports the **contract** + **registry**, NOT the other slice.

```ts
// shared/block-renderer/contract.ts
export interface BlockRenderer {
  type: string
  Component: ComponentType<{ blockId: string }>
}

// shared/block-renderer/registry.ts
const renderers = new Map<string, BlockRenderer>()
export function registerBlockRenderer(r: BlockRenderer): void
export function getBlockRenderer(type: string): BlockRenderer | null

// slices/editor/init.ts
registerBlockRenderer({ type: "database", Component: lazy(() => import("@/shared/block-renderer/loader/DatabaseLoader")) })

// shared/block-renderer/loader/DatabaseLoader.tsx — lazy-loads the database renderer
```

This breaks the static import cycle. Cross-slice rendering happens at runtime via registry.

#### Pattern 2 — `editor → block-selection` + `editor → comments` (7 edges)

Editor imports `shared/state/selection` instead of `slices/block-selection`. Block-selection slice becomes a **thin UI layer** over the shared state.

#### Pattern 3 — `admin-panel → templates` (5 edges)

Admin edits templates. Templates is a domain slice. Admin shouldn't have direct cross-slice import.

**Fix:** Templates slice exposes a public API via `slices/templates/index.ts` barrel. Admin imports from the barrel (allowed as `@/slices/templates` IFF whitelisted) OR templates exposes an admin-extension point.

Better: introduce `shared/admin/registry/` where slices that need an admin tab register themselves. Admin panel renders registry, never imports slice internals.

```ts
// shared/admin/registry.ts
registerAdminPanel({
  slug: "templates",
  label: "Templates",
  Component: lazy(() => import("./loaders/templates")),
})

// shared/admin/loaders/templates.tsx
// One-line lazy loader bridging the cross-slice gap.
export { TemplateEditor as default } from "@/slices/templates"
```

#### Pattern 4 — `workspace-sidebar → 6 slices` (6 edges)

**Fix:** Each slice contributes a nav entry via `defineFeature({ nav: { ... } })`. Sidebar consumes the **feature registry** (already exists in this project), iterates registered features, renders their nav metadata. Zero direct slice imports.
