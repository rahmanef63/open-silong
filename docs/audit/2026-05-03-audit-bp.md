FINDINGS — 12

1. **P1 — Cache Components disabled, no migration plan beyond a TODO comment.** `next.config.mjs:7-12` documents that `cacheComponents: true` is blocked by `ConvexAuthNextjsServerProvider` reading cookies dynamically. On Next 16 this is the modern caching model; staying off it on a Postgres-backed self-hosted deploy means every dynamic render hits Convex on cold cache. Fix: move auth provider behind a per-route Suspense boundary in `app/layout.tsx`, then enable `cacheComponents` and add `"use cache"` + `cacheTag` on `/share/[id]/page.tsx` (the only currently-anonymous reachable route). Doc: Next 16 Cache Components (`nextjs.org/docs/app/api-reference/config/next-config-js/cacheComponents`).

2. **P1 — `typescript.ignoreBuildErrors: true` in production build.** `next.config.mjs:16` and `tsconfig.json:11,26,29` (`strict:false`, `noImplicitAny:false`, `strictNullChecks:false`). The TODO acknowledges this is legacy-slice debt. Risk: every Next build silently ships type errors; React 19 ref-as-prop / async params / `use()` migrations land without compiler help. Fix (incremental): turn `strictNullChecks:true` on per-folder via project references, gate `next build` behind `tsc --noEmit` in the GitHub Action, drop the flag once `frontend/legacy-app` is removed.

3. **P1 — Multi-instance readiness incomplete on a single-replica deploy that may scale.** `next.config.mjs` does NOT set `cacheHandler` and the dockerfile/compose doesn't pin a shared cache (Redis / S3). `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` IS documented in `.env.example` (good), but no Server Actions exist in the repo today (so missing-key risk is latent, not active). If/when scaled to 2+ Next replicas behind Dokploy, ISR + Server Actions encryption will diverge. Fix: document the single-replica assumption explicitly in `README.md` runbook OR wire a shared `cacheHandler` before raising `replicas: > 1`. Doc: Next self-hosting docs (`nextjs.org/docs/app/building-your-application/deploying`).

4. **P2 — 3 raw `<a href>` for in-page anchors on the marketing page.** `app/page.tsx:56,59,62` — `#features`, `#how`, `#faq`. These are intra-document jumps so technically not a `next/link` violation (no route change), but the consistency check catches them. Fix: leave as-is and add a script-level allowlist comment OR convert to `next/link` (works fine for hash links).

5. **P2 — `dangerouslySetInnerHTML` used on hand-typed copy with HTML entities.** `app/page.tsx:193,264,319,368,425` — pattern `dangerouslySetInnerHTML={{ __html: it.q }}` for FAQ/testimonials/etc. The strings are author-controlled so XSS risk is low, but the pattern normalises an unsafe primitive into the codebase and breaks accessibility / streaming. Fix: replace `&lsquo;`/`&rsquo;`/`&apos;`/`&ldquo;`/`&rdquo;` literals with their actual unicode characters (`'`, `'`, `"`, `"`) in source and render via plain JSX.

6. **P2 — 1 raw `<img>` in DatabaseGalleryView.** `frontend/slices/databases/views/GalleryView.tsx:100` — covers user-supplied URL or data: URI. Fix: switch to `next/image` with `unoptimized` (since the URL is outside the Next allowlist) or keep raw `<img>` but add `loading="lazy"` + explicit `width`/`height` to prevent CLS.

7. **P2 — 1 `<img>`-equivalent path through `Image` already correct in AuthForm.** `app/auth/AuthForm.tsx:4` — uses `next/image` correctly. Listed for completeness.

8. **P2 — Mobile UX: no bottom navbar on a PWA-first product.** Current shell (`frontend/legacy-app/AppShell.tsx`) gives the mobile user only a hamburger trigger + search button at the top. For a PWA installed to home screen, the standard pattern is a fixed bottom tab bar (Home / Inbox / New / Search / Profile) with `env(safe-area-inset-bottom)` padding. Fix scope: new `frontend/slices/mobile-nav/components/BottomNav.tsx` rendered by `AppShell.tsx` only when `useIsMobile()` is true; hide on `/share/[id]` and `/auth`.

9. **P2 — Hydration mismatch risk in `useIsMobile`.** `frontend/shared/hooks/use-mobile.tsx:6` initializes state to `undefined` then sets after `useEffect`. The `Sidebar` component renders the desktop tree on first paint (server) but may flip to the mobile sheet on hydration if viewport is narrow. Acceptable today (no SSR for the dashboard since `ssr:false`), but if/when `cacheComponents` is enabled and any mobile-aware component renders SSR, this flips to a real hydration-mismatch. Fix: read `userAgent.isMobile` from the `headers()` API in a server boundary, or accept a `defaultIsMobile` prop seeded from the proxy.

10. **P2 — `app/page.tsx` Footer renders `{new Date().getFullYear()}` directly.** `app/page.tsx:489` — server-rendered, fine for now. With `cacheComponents` on, this becomes a per-request dynamic primitive. Fix: lift to a constant `2026` or wrap in a small client component if the year matters.

11. **P2 — Sidebar opens to `defaultOpen=true` regardless of cookie at SSR time.** `frontend/shared/ui/sidebar.tsx:51` — the cookie `sidebar:state` is set on every toggle, but the SSR pass doesn't read it. Result: every refresh on mobile flashes the desktop tree before flipping. Fix: in `app/dashboard/[[...slug]]/page.tsx`, read `cookies().get("sidebar:state")` and pass `defaultOpen` down. (Currently moot because the dashboard is `dynamic ssr:false`; revisit when migrating off legacy SPA.)

12. **P1 — Dashboard route is `next/dynamic` with `ssr:false`.** `app/dashboard/[[...slug]]/ClientLegacy.tsx:5-8` — the entire workspace is a client SPA wrapped in a Next route. This explicitly forfeits Next 16's RSC, streaming, partial pre-render, and Cache Components benefits. Documented as legacy-slice migration debt. Fix: track in roadmap; convert `Inbox`, `Trash`, `Settings`, `Profile` routes to native App Router segments (each getting their own `loading.tsx` / `error.tsx`) before turning Cache Components on.

FINDINGS — 5

14. **P1 — `.collect()` without index on whole tables in admin queries.** `convex/admin/queries.ts:29-36` (8 tables in one `Promise.all`) and `:61, :86, :88` plus `convex/templates/mutations.ts:83`. These are admin-gated by `requireAdminQuery`, but the dataset grows linearly with platform usage. At 10k users * 100 pages each = 1M rows scanned per overview hit. Fix: persist counters in a `metricsAggregates` table updated by an `internalMutation` triggered from the relevant write paths, or use Convex's `aggregate` component. Doc: Convex "Reading data" + "Aggregations" components.

15. **P1 — Sub-resource queries `.collect()` per page.** `convex/features/comments/queries.ts:23,45` returns ALL comments for a page (no `.take()` cap). A page with 5k comments hits the per-query 8MB limit. Fix: add `.take(200)` + paginate via cursor.

16. **P2 — Inbox `markAllRead` walks every unread notification then `patch`es one at a time.** `convex/features/inbox/mutations.ts:53-58`. Convex commits each `patch` atomically inside the mutation, but at >1k unread items this risks the 4s mutation budget. Fix: bound to N=500 and require a second call to drain, OR schedule an `internalMutation` to continue.

17. **P2 — Author identity check on the super-admin path uses email.** `convex/_shared/auth.ts:67-73` — `requireSuperAdmin` reads `user.email` from the `users` table and compares to `SUPER_ADMIN_EMAIL` env. The `users.email` is server-minted by Convex Auth (not raw `identity.email`), so this is acceptable defense-in-depth, but the rule says prefer `subject` / explicit `userId` allowlist. Fix: store a flag on `userProfiles.role === "super_admin"` set during a one-time bootstrap, then drop the email comparison.

18. **P2 — `getPublicShare` uses `as Id<"pages">` cast on a raw client string.** `convex/pages.ts:21` wraps `ctx.db.get(id as Id<"pages">)` in try/catch. Works, but Convex 1.36 ships `ctx.db.normalizeId("pages", id)` which returns `null` instead of throwing. Fix: replace try/catch with `normalizeId` for cleaner control flow.

### New findings (cycle 2)

- **DELTA-SEC-IDENTITY-001** — high (P0). `convex/_shared/auth.ts:62-72`
  (`requireSuperAdmin`) authorises by **lowercased email match** against
  `SUPER_ADMIN_EMAIL`. `@convex-dev/auth` Password provider lets the same
  email be (re)bound to a different `users._id` if the bootstrap row is
  deleted/recreated, and `users.email` is verifier-trusted, not
  cryptographic identity. Per the project's CLAUDE.md security rule
  ("Convex public fns: permission check before any DB write") and the
  audit-bp Convex rule "never key auth off `identity.email`; use
  `subject` or a server-minted user record" — switch to a `userProfiles`
  role check or pin to `users._id` from a one-shot bootstrap mutation.
  Currently low blast-radius (no public callers found that hit
  `requireSuperAdmin` directly in the cycle-2 diff), but it is the
  `requireAdmin`-bypass path on the existing audit log + role
  surfaces — keep at P0.

- **DELTA-CONVEX-COMMENTS-PUBLIC-WRITE-001** — high (P0).
  `convex/features/comments/mutations.ts:7-39` (`create`) authorises a
  comment write whenever the parent page is `isPublic`, but the only
  identity check it performs on the actor is `getAuthUserId(ctx)` —
  meaning **any signed-in Nosion user can post comments on any other
  user's public page**, and `update`/`resolve`/`remove` (lines 41-78)
  let the comment author do anything to it forever. There is no
  rate-limit, no abuse mute, no page-owner moderation surface. For a
  shared-link product this is a spam vector and a defacement vector
  (comment text rendered alongside the page). Pair with a
  `@convex-dev/rate-limiter` token bucket scoped per
  `(actorId, pageId)` and add a `comments.moderate` mutation gated on
  `page.userId === actorId`.

- **DELTA-CONVEX-VALIDATOR-INBOX-KIND-001** — medium (P1).
  `convex/features/inbox/mutations.ts:8` (`create`) accepts
  `kind: v.string()` but the schema comment in `convex/schema.ts:96-97`
  enumerates `"mention" | "comment" | "share" | "system" | "update"`.
  Convex docs treat validator tightness as the security boundary —
  switch to `v.union(v.literal("mention"), …)` so the schema-level
  contract is actually enforceable on the wire. Same pattern applies
  to the `kind`/`target` fields on `convex/admin/mutations.ts:73-78`
  (`logAuditEvent`).

- **DELTA-CONVEX-SHARE-COMMENT-LEAK-001** — medium (P1).
  `convex/features/comments/queries.ts:19,41` returns comments to
  anyone (`getAuthUserId` returns null is fine — but with
  `page.isPublic` true, it returns to *signed-in callers other than
  the owner*). Combined with `getPublicShare` being anonymous-readable
  but `listForPage` requiring sign-in, comments aren't visible on the
  public share view at all — the gate is ineffective at its stated
  purpose AND leaks to peer users who shouldn't see threads. Either
  drop the comment-on-public-page surface entirely or wire a separate
  anonymous reader DTO.

- **DELTA-NEXT-DASHBOARD-ALL-CLIENT-001** — medium (P1).
  `app/dashboard/DashboardShell.tsx:1` is `"use client"` and wraps
  every dashboard route in a single client tree (sidebar + auth guard
  + theme + tooltip + toaster). Cycle-2 reinforced this by folding
  more state up (`searchOpen`, CRUD dialogs). Per Next 16 server-first
  guidance + the project's deferred Cache Components plan in
  `docs/audit/cache-components.md`, this is the structural reason
  `cacheComponents: true` cannot be flipped — `ConvexAuthNextjsServerProvider`
  is fine in `app/layout.tsx`, but the auth guard is doing client-side
  redirect that proxy.ts already handles. Splitting `DashboardShell`
  into `<DashboardShellServer>` (layout chrome, server) + slim
  `<DashboardClient>` islands unblocks the Cache Components migration.

- **DELTA-CONVEX-DUP-AUTH-PATTERN-001** — low (P2). 22 of 22 mutations in
  `convex/pages.ts`, `convex/databases.ts`, `convex/snapshots.ts`,
  `convex/recents.ts`, `convex/preferences.ts`,
  `convex/features/inbox/mutations.ts`,
  `convex/features/comments/mutations.ts`,

- **DELTA-PERF-CHAT-NO-AUTH-001** — medium (P1).
